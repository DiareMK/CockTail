# Звіт до Практичної роботи №6 (Налагодження коду)

## 1. Процес виявлення та виправлення помилок

Під час роботи було змодельовано та виправлено кілька типів поширених помилок:

### А) Помилка типів (Type Error) у стані лайків
- **Проблема:** Використання `setLikes(likes + '1')` призвело до того, що число 0 перетворилося на рядок "01", "011" тощо.
- **Виявлення:** За допомогою **React DevTools** (вкладка Components) було помічено, що стан `likes` має тип String замість Number.
- **Виправлення:** Змінено на `setLikes(likes + 1)`.

### Б) Попередження про ключі (Key Warnings)
- **Проблема:** Використання статичного ключа `key="duplicate-key"` для всіх елементів списку.
- **Виявлення:** Консоль браузера (F12) вивела попередження: "Warning: Each child in a list should have a unique 'key' prop".
- **Виправлення:** Повернуто використання унікального ідентифікатора `{drink.idDrink}`.

### В) Логічна помилка у фільтрації
- **Проблема:** Використання `&&` замість `||`. Фільтр вимагав, щоб назва збігалася з пошуком і одночасно коктейль належав до певної категорії, що робило пошук занадто жорстким.
- **Виявлення:** Пошук не видавав результатів, хоча дані були присутні. Перевірка логіки у вкладці **Sources** (через Breakpoint) показала, що умова завжди повертає false.
- **Виправлення:** Заміна логіки на правильну або використання одного критерію.

### Г) Нескінченний рендер (Too many re-renders)
- **Проблема:** Виклик `onClick={setLikes(likes + 1)}` (без анонімної функції).
- **Виявлення:** Додаток "впав" з помилкою у консолі: "Too many re-renders. React limits the number of nested updates...".
- **Виправлення:** Перетворення у функцію зворотного виклику: `onClick={() => setLikes(likes + 1)}`.

## 2. Відповіді на контрольні запитання

**1) Перевага React DevTools над звичайним інспектуванням DOM?**
React DevTools дозволяє бачити структуру компонентів, їхні **props** та **state**, а також дерево Virtual DOM. Звичайний інспектор показує вже відрендерений HTML, де не видно логіку React та дані, що керують інтерфейсом.

**2) Відловлювання неоптимального рендерингу через ключі?**
Якщо ключі не унікальні або змінюються при кожному рендері (наприклад, `Math.random()`), React не може ефективно перевірити, які елементи змінилися, і перерендерить весь список. Це помітно через спалахи в інспекторі або через вкладку **Profiler** у DevTools.

**3) Алгоритм роботи з Breakpoint:**
1. Відкрити вкладку **Sources** у DevTools.
2. Знайти потрібний файл скрипта.
3. Натиснути на номер рядка, щоб встановити точку зупинки.
4. Відтворити дію у програмі - виконання зупиниться на цьому рядку.
5. Проаналізувати значення змінних у цей момент.

**4) Ключове слово для зупинки виконання коду:**
Ключове слово `debugger;`. Воно працює як автоматичний Breakpoint, якщо інструменти розробника відкриті.

**5) Значення дій у дебагері:**
- **Step Over:** Перейти до наступного рядка коду, не заходячи всередину функцій.
- **Step Into:** Зайти всередину функції, що викликається на поточному рядку.
- **Step Out:** Вийти з поточної функції та повернутися до коду, який її викликав.
